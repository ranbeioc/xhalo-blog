# Production Gate Audit

This document acts as the formal quality and security gate review for integrating `xhalo-blog` with the production environment. 

---

## Current Status: GATED / BLOCKED

**Production integration is not approved.** 

Before any production keys, repositories, or assets are connected, the system must remain strictly at **Level 0 (Hold)**.

---

## 1. Requirements Before Promotion to Level 1 (Read-only Verification)

Level 1 enables the headless dashboard to read content and verify structure compatibility with the production Hexo blog, without write permissions.

- [ ] **Codebase Consistency**:
  - `README.md` must accurately reflect current capabilities, including the status of Phase 7.1 Queue Worker async publishing.
- [ ] **Verification Documentation**:
  - `docs/staging-verification-evidence.md` must be populated with verified staging outcomes using the Phase 7.1 async queue loop.
- [ ] **Header and Token Naming**:
  - Ensure all documentation only refers to headers supported by the API code (`x-xhalo-admin-secret` and `cf-turnstile-token`).
- [ ] **Execution Reports**:
  - PR execution reports for historical core milestones must be complete.
- [ ] **Static Scanners**:
  - Regex patterns in `scripts/check-no-production-markers.mjs` must be correct and verified.
  - `npm run check:all` must pass cleanly.
- [ ] **No Open P0/P1 Documentation Mismatches**:
  - No claims of features being implemented that are only scaffolded.

---

## 2. Requirements Before Promotion to Level 2 (PR Generator Mode Trial)

Level 2 allows the dashboard to commit draft changes to isolated branches and open Pull Requests.

- [ ] **Level 1 Complete**:
  - Level 1 read-only integration has run successfully in production for at least 7 days with zero parser errors or database index crashes.
- [ ] **Staging Async Publish Fully Verified**:
  - Staging async publish execution has passed all 17 cases in `docs/async-publish-verification-matrix.md` with logs verified.
- [ ] **GitHub App Permissions Configured**:
  - Minimal repository permissions (Contents: Read/Write, Pull Requests: Read/Write, Metadata: Read-only) verified on the production repository.
- [ ] **Branch Protection Configured**:
  - Protection rules on `main` are enabled (direct pushes blocked, 1+ approving review required, linear history required).
- [ ] **No Auto-Merge**:
  - Verified that auto-merging of pull requests generated by the GitHub App is disabled in the codebase and blocked by branch policies.
- [ ] **No Direct Main Pushes**:
  - Configured GitHub App settings to restrict pushing directly to the `main` branch.
- [ ] **Disaster Recovery Verified**:
  - The database backup/restore runbook has been dry-reviewed and tested in staging.
- [ ] **Wrangler Environment Variables**:
  - `LIVE_WRITES_ENABLED=false` remains the default state in all environments and config repositories.
- [ ] **Manual Approval Gate**:
  - Explicit written approval from the repository owner is required to promote to Level 2.

---

## 3. Strict Prohibitions

The following behaviors are strictly prohibited and will invalidate the gate audit:

* **Direct default branch (`main`) writes** from the worker context.
* **Auto-merging** of generated PRs.
* **Public unauthenticated mutation endpoints**: all `/api/*` endpoints must be guarded by Cloudflare Access or the validated fallback admin header.
* **Production secrets** committed in documentation, configuration templates, or code comments.
* **Production database destructive schema changes** (all migrations must be backward-compatible and non-destructive).
